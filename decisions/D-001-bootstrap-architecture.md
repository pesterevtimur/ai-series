# D-001 — Bootstrap-архитектура проекта

## Контекст

Это первое архитектурное решение проекта `Auto-ai-series` — анимационного сериала об эволюции ИИ как психологической и социальной драмы (см. конституцию автора, передана как первое сообщение сессии). Решения этого документа фиксируют развилки, выбранные в brainstorming-сессии 2026-05-24, перед записью полного design-спека ПП1.

Все решения здесь либо реально были спорными (>2 структурно разных опции), либо влияют на CLAUDE.md законы — то есть подпадают под порог "значимая развилка" из конвенции `decisions/`.

---

## Базовые решения (5)

### B-1. Внешние агенты в системе

**Опции:** Гибрид (Claude Code primary + точечно Codex/Desktop) / Triada как в leadecho (Desktop=архитектор, Code=writer, Codex=ревьюер) / Single Claude Code.

**Принято:** Гибрид без Codex.
- **Claude Code** — primary исполнитель (шоураннер, все скиллы, все 6 субагентов-критиков)
- **Claude Desktop** — периодическая проверка папки через filesystem-MCP (read-only)
- **Claude Cowork (Claude.ai web)** — внешний ресёрч-собиратель для golden data set, пишет только в `raw/_cowork-dumps/` через копирование Artifacts вручную
- **Codex** — НЕ используем (по запросу Тимура: "по коду лучше справляется, тут не нужен")

**Следствие:** AGENTS.md не требуется (Codex выключен). Адверсариальная критика полностью переезжает на ВНУТРЕННИЕ субагенты Claude Code — внешнего "критика" нет, поэтому A5·PHILOSOPHY должна быть особенно жёсткой.

---

### B-2. Парадигма внутренних агентов (6 нарративных ролей)

**Опции:** Гибрид (шляпы + изолированные субагенты на критике) / Все через шляпы / Все через реальные Task-subagents.

**Принято:** Гибрид.
- Рутинные проходы (драфты, переключение перспектив) — через "шляпы" в одной Claude Code сессии
- Адверсариальная критика (A5 PHILOSOPHY, A3 INCENTIVE, A4 DIALOGUE voice-check, A1 LORE realism, A2 CHARACTER truth, A6 AUDIENCE boredom) — через **реальные изолированные субагенты** в `.claude/agents/<name>.md` с собственным system prompt, своим контекстом и доступом ТОЛЬКО к Read/Grep/Glob (READ-ONLY, никакого Edit/Write)

**Следствие:** Структурная гарантия независимости критиков. Критик не может "помочь" переписать сцену — может только указать на проблему через структурированный отчёт.

---

### B-3. Знаниевый слой (контроль consistency)

**Опции:** Markdown + lint-скрипт consistency-check / Wiki-слой с citations (как agentic-ops-cc) / Только Story Bible + reactive voice-check.

**Принято:** Markdown + lint-скрипт.
- `lore/`, `characters/`, `story-bible/`, `episodes/`, `scenes/` — обычные md-файлы с обязательным frontmatter (`id`, `version`, `status`, `references`)
- `tools/consistency_check.py` сверяет факты артефакта против Bible / character-sheets через frontmatter references
- Без wiki/citation/raw-layer accountability (за исключением `raw/_cowork-dumps/` как immutable inbox от Cowork)

**Следствие:** Лёгкий вход в систему, анти-дрифт работает через формальную валидацию.

---

### B-4. Язык артефактов

**Опции:** Всё на русском / Гибрид (творчество рус., инфра англ.) / Всё на английском.

**Принято:** Всё на русском.
- Story Bible, lore, characters, episodes, SKILL.md, system prompts субагентов, CLAUDE.md, lint-сообщения, аннотации в golden — на русском.
- **Исключение:** Superpowers skills используются на их родном языке (английский, без перевода), цитаты из источников в golden — на языке оригинала. См. Language Policy в design-спеке для полной формулировки.

**Следствие:** Тон и нюанс творческого продукта сохраняются. Чтение английских Superpowers скиллов — accepted overhead.

---

### B-5. Scope ПП1 (где остановиться)

**Опции:** Инфра + smoke-test на мини-материале / Только инфра, dummy-тесты / Слияние ПП1+ПП2 (инфра + полный Story Bible v0.1).

**Принято:** Инфра + smoke-test на мини-материале.
- ПП1 строит: репо, скиллы, субагентов, lint-скрипты, lineage, golden процесс
- Smoke-test проверяет систему end-to-end на мини-материале (тезис + 5 правил мира + 1 character) из конституции
- Story Bible v0.1 — отдельный ПП2

**Следствие:** Минимальный risk-by-validation — мы убеждаемся что система работает на живом материале до того как тратим усилия на полный Bible.

## Правки из self-review (4)

После презентации 8 секций дизайна автору, Claude Code провёл адверсариальный self-review всех секций против целей проекта. Найдено 10 находок (4 критичных, 4 важных, 2 операционных). Тимур одобрил все 4 критичных правки.

### R-1 (от D1). Дивергентная generation в draft-скиллах

**Проблема:** Заложена мульти-агентная КРИТИКА, но не мульти-агентная ГЕНЕРАЦИЯ. Шоураннер (одна Claude Code сессия) генерирует все draft'ы в одном контексте — это прямой провал анти-гомогенизации (раздел 5 конституции).

**Принято:** `draft-story-bible-section` и `draft-character-sheet` запускают **3 параллельных Task-вызова** с разными доминантными уклонами (LORE-dominant, PHILOSOPHY-dominant, CHARACTER-dominant). Шоураннер собирает 3 структурно разных варианта, явно выбирает или синтезирует. Cost-impact: +3× токенов на drafting, но это и есть смысл дивергенции.

---

### R-2 (от D3). Enforcement реакции на critic concerns

**Проблема:** Workflow говорит "шоураннер реагирует на concern/veto", но это рекомендация в тексте, не enforced правило. Шоураннер = единая точка отказа в записи; может молча проигнорировать concern.

**Принято:** Добавлен **6-й lint-скрипт `tools/concern_resolution_validator.py`**. Для каждого concern/veto в отчёте `adversarial-review-pass` требует либо (а) commit-diff артефакта, явно адресующий concern, либо (б) запись в `decisions/D-NNN-*.md` с обоснованием "оставляю, потому что...". Без этого commit блокируется через `evidence-before-action`.

---

### R-3 (от D4+D7). Regression set и regression-пары для критиков

**Проблема:** Pressure-tests проверяют, что обойти discipline-скилл нельзя (процедура). Но НЕ проверяют, что результат процедуры (вывод критика) реально ловит плохой контент. Один character-sheet в smoke-test не калибрует A2 на сравнительную способность.

**Принято:** Создаётся **`tests/regression/`** с двумя видами материала:
- **Намеренно плохие артефакты** (story-bible со страуменом, character с морализаторством, сцена с voice bleed) — соответствующий критик должен выдать veto. Не выдал = не калиброван, ПП1 не принят.
- **Regression-пары "сильный + слабый"** для каждого из 6 критиков — verdicts должны разойтись. Не разошлись = критик не различает.

Это **дополнение к pressure-tests**, не замена.

---

### R-4 (от D10). Token cost commitment

**Проблема:** 6 субагентов на Opus + extended thinking × повторные вызовы при failed validation × передача контекста = десятки тысяч токенов на каждый артефакт. Это flagged constraint.

**Принято:** Оставляем "все 6 на Opus + ext.thinking" как решено в Section 4. **Явный cost commitment** фиксируется в спеке и в Section 11 (flagged risks). Альтернатива (Opus для A5/A3/A1, Sonnet для A2/A4/A6) отклонена — Тимур приоритет качества над стоимостью.

## Важные находки (флагованы в спек, не правка дизайна, обсуждаемы позже)

- **D2.** Cowork собирает по нашим запросам → корпус отражает bias шоураннера+Тимура. Митигация: в `golden/README.md` зафиксировано правило **golden-audit** раз в квартал — Cowork ищет материалы, противоречащие нашему текущему пониманию "сильно".
- **D5.** Language policy явно прописана в Section 10 спека.
- **D6.** Golden subjective by nature — flagged в README.
- **D8.** Operational procedures (определения "сессии", процедуры log.md / master-plan / decisions thresholds) вынесены в новую **Section 8.5** спека.
- **D9.** Multi-agent operational complexity (Cowork↔Code↔Desktop ручная координация) flagged как operational risk на будущее. Митигация в ПП2+: возможно скилл `session-bootstrap`.

## Открытые технические вопросы (на стадию writing-plans)

1. **Механизм extended thinking в `.claude/agents/<name>.md`** — поддерживается ли явно в frontmatter (например, `thinking_budget: max`) или только через инструкцию в system prompt ("ultrathink перед каждым выводом"). Проверить в implementation phase.
2. **Конкретный SHA Superpowers для pinning** — берём `f2cbfbef` от agentic-ops-cc, но если выйдет более свежий критичный release до старта implementation — переоценить.
3. **TF-IDF пороговое значение для voice_dissimilarity.py** — стартовое 0.65 (эвристика), калибровать на regression-парах из R-3.

## Что НЕ решено в этом D-001 (откладывается)

- Структура и состав сезона 1 (это ПП5)
- Конкретный состав кор-каста (это ПП3)
- Выбор центрального тезиса в финальной форме (это ПП2)
- Выбор рабочего названия сериала (после Story Bible v0.1, как в конституции)
- Дизайн ИИ — стратегия из 4 опций (это ПП3, после кор-каста)
